# SKILL TRANSCRIPT — แพลนเดโม่ (Owl Day House × ม.พะเยา)

ระบบจำลอง "ทรานสคริปต์ทักษะ" — กิจกรรม → คะแนน 6 ด้าน → เรดาร์ใยแมงมุม
อัปเดต: ก.ค. 2026 · สถานะ: แพลนพร้อมส่ง Claude Code

---

## 0. TL;DR (อ่าน 30 วิ)

- **เป้า:** ทำเดโม่สวย ๆ ไปปิดโปรเจกต์กับ ม.พะเยา ต่อจากที่เสนอไปหลายอย่าง
- **แก่น:** เด็กเข้าร่วมกิจกรรมคณะ → แต่ละกิจกรรมให้คะแนน 6 ด้าน (Knowledge / Skills / Attitude / Ethics / Aesthetics / Wellness) → พอจะจบก็สรุปเป็น **เรดาร์ 6 แกน** บอกว่าเด่นด้านไหน
- **ขอบเขตเดโม่:** จำลองระบบกลางแยกออกมา (ไม่ต่อ SSO/ทะเบียนจริง) รองรับ 3 อย่าง: **สร้างกิจกรรม · นิสิตสแกนเข้าร่วม · dashboard กิจการนิสิต**
- **โมเมนต์ขาย:** สแกนมือถือ → ชื่อเด้งขึ้นจอสด ๆ + หน้า transcript เรดาร์สวยตอนปิดการนำเสนอ
- **สถาปัตย์:** React (Vite) frontend-first + **store แบบสลับได้** (localStorage ก่อน → backend จิ๋วเสียบตอน pitch) · ต้นทุน ~0

## 1. ทำไมต้องทำเดโม่นี้ (เดโม่นี้ต้องพิสูจน์อะไร)

นี่คือ critical path "มี demo สวยอวด → รับงาน" ตรง ๆ กับลูกค้าที่มีอยู่แล้ว (ม.พะเยา = สายเครดิตราชการ/มหาลัย = moat ของ ODH) เดโม่ต้องทำให้กรรมการ/อาจารย์เห็นภาพ 3 อย่างภายใน 5 นาที:

1. **มันง่ายสำหรับเด็ก** — สแกนแล้วจบ ไม่ต้องกรอกอะไรวุ่นวาย
2. **มันง่ายสำหรับเจ้าหน้าที่** — สร้างกิจกรรม + ดูใครมาแล้วบ้าง แบบเรียลไทม์
3. **มันมี "ของ" ที่ระบบอื่นไม่มี** — Transcript เรดาร์ที่เล่าเรื่องเด็กได้ทั้งคน ใช้ให้อาจารย์ที่ปรึกษา/ฝ่ายพัฒนานิสิตต่อยอดได้

> เดโม่นี้ **ไม่ใช่** ระบบ production — เป็นเครื่องมือปิดการขาย ทำแค่พอให้เชื่อและตื่นเต้น อย่าไป build ระบบจริงทั้งก้อน (กับดักตามข้อ 10 ของ PROJECT_CONTEXT)

## 3. ขอบเขต (สำคัญมาก — กันงานบาน)

### ✅ อยู่ในเดโม่
- สร้าง/แก้/ดูกิจกรรม + ตั้งน้ำหนักคะแนน 6 ด้าน
- QR / join code ต่อกิจกรรม
- นิสิตสแกน/กรอกโค้ด → เข้าร่วม → ชื่อเข้ารายการ (เรียลไทม์)
- รายชื่อผู้เข้าร่วมต่อกิจกรรม (อัปเดตสด)
- หน้า Transcript นิสิต = เรดาร์ 6 แกน + จุดเด่น/ควรเสริม + รายการกิจกรรม
- Dashboard กิจการนิสิต: KPI + กราฟภาพรวม + ค้นหานิสิต + drill-down
- Seed data สมจริง (นิสิต/กิจกรรม/การเข้าร่วม เยอะพอให้ทุกหน้ามีเนื้อ)
- ปุ่ม reset/seed สำหรับ demo ซ้ำ

### ❌ ไม่อยู่ในเดโม่ (พูดปากเปล่าว่า "ต่อได้ในเฟสจริง")
- ต่อ SSO / ทะเบียนนิสิตจริง / ระบบกลางจริงของ มพ. → **จำลองแทน**
- Login/สิทธิ์จริง (เดโม่ใช้ "เลือก role/เลือกคนที่เป็น" ไม่มีรหัสผ่าน)
- อนุมัติกิจกรรม/เวิร์กโฟลว์หลายขั้น, การออกใบจริง, การชำระเงิน
- แอปมือถือ native (ใช้ web mobile-first แทน "แอปมหาลัยสมมุติ")
- ระบบแจ้งเตือน push, รายงานส่งออก PDF (ทำได้ทีหลัง)

## 4. Actors & แกนคะแนน 6 ด้าน

**Actors**
| Actor | ทำอะไรในเดโม่ |
|---|---|
| นิสิต | สแกนเข้าร่วม, ดู transcript ตัวเอง |

**6 ด้าน** (แต่ละกิจกรรมให้คะแนนแต่ละด้าน 0–10)

## 5. User Flows (ละเอียด)

### Flow A — เจ้าหน้าที่สร้างกิจกรรม
1. เข้าหน้า organizer → กด "สร้างกิจกรรม"
2. กรอก: ชื่อ, คณะผู้จัด, วันเวลา, สถานที่, คำอธิบาย, ความจุ (ไม่บังคับ)
3. **ตั้งคะแนน 6 ด้านด้วยสไลเดอร์** → ข้าง ๆ โชว์ **มินิเรดาร์ preview** ของ "รูปทรงคะแนน" ที่กิจกรรมนี้จะให้ (เห็นทันทีว่ากิจกรรมนี้ "ทรงไหน")
4. บันทึก → ระบบสร้าง **join code + QR** → กิจกรรมสถานะ `open`
5. เด้งไปหน้า activity detail (มี QR ใหญ่ไว้ฉายจอ)

### Flow B — นิสิตสแกนเข้าร่วม (พระเอกฝั่งง่าย)
1. นิสิตเปิด "แอปมหาลัยสมมุติ" (web mobile) → กดสแกน (หรือกรอก join code)
2. สแกนได้กิจกรรม → **เดโม่: เลือกว่าตัวเองเป็นใคร** (dropdown/ค้นชื่อ เพราะไม่มี SSO จริง)
3. เห็นการ์ดกิจกรรม + ปุ่ม "ยืนยันเข้าร่วม" → กด
4. ✅ Success + แอนิเมชันเบา ๆ + preview ว่า "กิจกรรมนี้เติมด้านไหนให้คุณ"
5. ชื่อเด้งเข้ารายการผู้เข้าร่วม (ฝั่ง organizer เห็นสด) + คะแนน snapshot เข้า transcript

### Flow C — Transcript ของนิสิต (พระเอกฝั่ง "ของ")
1. นิสิตเปิดหน้า "ทรานสคริปต์ของฉัน"
2. **เรดาร์ 6 แกน** วาดด้วยแอนิเมชัน (ดึงจากผลรวมคะแนนทุกกิจกรรมที่เข้าร่วม)
3. "🌟 จุดเด่น: …" (2 ด้านคะแนนสูงสุด) + "ควรเสริม: …" (2 ด้านต่ำสุด) + คะแนนสมดุล (balance score)
4. ไทม์ไลน์/รายการกิจกรรมที่เข้าร่วม (แต่ละอันโชว์ว่าให้ด้านไหนบ้าง)
5. **(optional) ฉายา Character AI** — ดูข้อ 12

### Flow D — Dashboard กิจการนิสิต
1. **ภาพรวม:** KPI (นิสิตที่มีส่วนร่วม, กิจกรรมทั้งหมด, เช็กอินรวม, เฉลี่ยกิจกรรม/คน)
2. กราฟ: การเข้าร่วมตามช่วงเวลา · การกระจาย 6 ด้านทั้ง cohort (ด้านไหนคณะจัดเยอะ/น้อย) · กิจกรรมยอดนิยม · แยกตามคณะ/ชั้นปี
3. **ตารางนิสิต:** ค้นหา → คลิกเข้า transcript รายคน
4. **มุมมองกิจกรรม:** ด้านไหน "under-served" (ทั้งมหาลัยขาด) → ใช้แนะนำให้คณะจัดเพิ่ม (ขายจุดนี้ให้กรรมการ)

## 6. Data Model

```
Faculty        id, name, name_en
Student        id, student_code, title, first_name, last_name,
               faculty_id, major, year, enrolled_year, avatar_url?
Activity       id, code, name, description, faculty_id, category?,
               start_at, end_at, location, capacity?,
               dims { knowledge, skills, attitude, ethics, aesthetics, wellness },  // 0–10 ต่อด้าน
               join_code, status(draft|open|closed), created_at
Participation  id, student_id, activity_id, checkin_at, status(checked_in|completed),
               dims_snapshot { … }   // ⚠️ สแนปช็อตคะแนนตอนเข้าร่วม กันแก้กิจกรรมย้อนหลังกระทบ transcript
```

**การคิดคะแนน transcript (ต่อ 1 นิสิต):**
- คะแนนด้าน X = ผลรวม `dims_snapshot.X` ของทุก participation ที่ `completed`
- เรดาร์ = คะแนน 6 ด้าน โดยตั้ง max ของแกน = ค่าสูงสุดใน cohort ต่อด้าน (ให้รูปทรงเทียบกันได้)
- **จุดเด่น** = top 2 ด้าน, **ควรเสริม** = bottom 2 ด้าน
- **Balance score** = ความสม่ำเสมอของ 6 ด้าน (เช่น 100 − ค่าเบี่ยงเบนที่ normalize แล้ว) → เล่าเรื่อง "รอบด้าน vs เชี่ยวเฉพาะทาง"

## 7. หน้าจอ (screens)

**Organizer**
- `Login/role select` — เลือกบทบาท (ไม่มีรหัสผ่านในเดโม่)
- `Activities list` — สถานะ + จำนวนคนเข้าร่วม

- `Seed/Reset` — ปุ่มรีเซ็ต/โหลดข้อมูลตัวอย่าง สำหรับเดโม่ซ้ำ

## 8. สถาปัตยกรรม & Tech (ข้อตัดสิน)

**Stack:** React + Vite + TypeScript · charting = Recharts *หรือ* Chart.js (radar) · routing = React Router · state = Zustand/Context เบา ๆ · QR = `qrcode` (สร้าง) + `html5-qrcode`/`@zxing` (อ่าน)

**หัวใจ: Data store แบบสลับได้ (store interface)**
```
interface Store {
  listActivities(), getActivity(id), createActivity(x), updateActivity(id,x)
  join(activityId, studentId), listParticipants(activityId)
  listStudents(), getStudent(id), getTranscript(studentId)
  dashboardStats()
  seed(), reset()
}
```
- **impl #1 = LocalStore** (localStorage) → ใช้เริ่มพัฒนา + เดโม่บนเครื่องเดียว/2 แท็บ (ต้นทุน 0, deploy Cloudflare Pages ได้)
- **impl #2 = ApiStore** (fetch → backend จิ๋ว) → **เสียบตอน pitch จริง** เพื่อให้ "สแกนมือถือ → เด้งขึ้นจอ" ทำงานข้ามเครื่อง
- UI เรียกผ่าน `Store` เท่านั้น → สลับ impl ได้โดยไม่แตะหน้าจอ

**Backend จิ๋ว (เฉพาะตอนต้องการโมเมนต์ข้ามเครื่อง)**
- Node + Express + `better-sqlite3` (หรือ lowdb/ไฟล์ JSON ก็พอ) · ~5–8 REST endpoints ตรงกับ `Store`
- รันบนเครื่องตัวเอง → มือถือต่อผ่าน **LAN IP** (วง wifi เดียวกัน) หรือ tunnel ชั่วคราว
- realtime แบบง่าย: ฝั่งรายชื่อ/dashboard poll ทุก 2–3 วิ (พอสำหรับเดโม่) — ไม่ต้องทำ websocket

**เลือกโหมดเดโม่**
| จะเดโม่แบบ | ใช้ store | หมายเหตุ |
|---|---|---|
| จอเดียว / 2 แท็บบนแล็ปท็อป | LocalStore | ง่ายสุด ปลอดภัยสุด (fallback) |
| มือถือจริงสแกน → เด้งขึ้นจอฉาย | ApiStore + backend จิ๋ว | โมเมนต์ wow ตัวจริง |

> คำแนะนำ: build ด้วย LocalStore ให้ครบก่อน แล้วค่อยเพิ่ม backend จิ๋วก่อนวันนำเสนอ 1–2 วัน — ไม่ over-build แต่เปิดทาง wow ไว้

## 9. Seed Data (ต้อง "สมจริง" — นี่คือสิ่งที่ทำให้ไม่ดูเป็น AI)

- ⚠️ **ใช้ข้อมูลปลอมล้วน** (กัน PDPA) — ถ้าจะเดโม่ด้วยชื่อนิสิตจริงในอนาคต ต้องขอ consent + แจ้งที่มา ตามกฎเหล็ก ODH

## 10. Design Direction (ใช้สูตรข้อ 7 ของ PROJECT_CONTEXT)

```
ไม่ดูเป็น AI = ข้อมูลจริง + signature ผูกโดเมน + ฟอนต์มีคาแรกเตอร์
             + SVG จริง + เลี่ยง 3 default look + motion เบา
```
- **Signature ของงานนี้ = เรดาร์/ใยแมงมุม + constellation** ให้เป็นลาย motif ทั้งระบบ (พื้นหลังจาง ๆ, ไอคอน 6 ด้านเป็น SVG สะอาด **ไม่ใช้ emoji**)
- **ฟอนต์:** display = Kanit (ไทย) / Space Grotesk · body = IBM Plex Sans Thai · ตัวเลข/โค้ด = IBM Plex Mono
- **สี:** ธีมม่วง–ทอง (มพ.) + สีประจำ 6 ด้านตามข้อ 4 (ใช้คงเส้นคงวาในเรดาร์/กราฟ/แท็ก)
- **เลี่ยง 3 ลุค default ของ AI:** ครีม+เซอริฟ+terracotta / ดำ+เขียวสะท้อนแสง / หนังสือพิมพ์เส้นบาง
- **Motion เบา:** เรดาร์วาดตอนโหลด, ชื่อผู้เข้าร่วม slide-in ตอนเด้งเข้า, ตัวเลข KPI count-up
- **บาร์คุณภาพอ้างอิง:** เทียบกับ `sinphattana-demo.html` (ข้อมูลจริงครบ + signature ชัด + พาเลตต์มีคาแรกเตอร์)

## 12. Character AI (optional / stretch — ขอคอนเฟิร์มความหมาย)

**การตีความ:** ให้ AI สรุปคะแนน 6 ด้าน → ออกมาเป็น **ฉายา/archetype + คำอธิบายสั้น** ปิดท้ายหน้า transcript (เช่น "นักสร้างสรรค์สายจริยธรรม — เด่นสุนทรียภาพ+คุณธรรม เหมาะกับงานที่ต้องใช้ทั้งใจและความคิด")

**ทำแบบเบา ๆ ในเดโม่ได้ 2 ทาง:**
- **Template-based (แนะนำสำหรับเดโม่):** map top-2 ด้าน → ฉายา + คำบรรยายที่เขียนไว้ล่วงหน้า → เร็ว, ไม่พึ่ง network, เดโม่นิ่ง
- **AI จริง (ถ้าอยากโชว์):** เรียก Anthropic API (มี `ANTHROPIC_API_KEY`) ส่งคะแนน 6 ด้าน → ขอ persona สั้น 2–3 ประโยค (บังคับ JSON, มี fallback เป็น template ถ้า API ล่ม)

> ทำเป็น feature แยกเปิด/ปิดได้ ไม่ให้บล็อกงานหลัก · **ถ้าพี่หมายถึงตัวละครที่โต้ตอบได้/แชตได้ บอกมา จะปรับ scope ใหม่**

## 13. สิ่งที่ต้องคอนเฟิร์ม (ก่อน/ระหว่างทำ)

1. **"Character AI"** = ฉายา/persona (ตามข้อ 12) หรืออย่างอื่น?
2. **สีทางการ มพ.** (ม่วง–ทอง?) + จะ theme เป็นของ มพ. หรือของ ODH? (แนะนำ: มพ. + credit ODH)
3. **โมเมนต์เดโม่** ต้องการ "มือถือจริงสแกนข้ามเครื่อง" ไหม (ถ้าใช่ = ต้องมี backend จิ๋ว M6)
4. **6 ด้านนี้ล็อกแล้วใช่ไหม** หรือ มพ. มีชุดสมรรถนะ/หมวดกิจกรรมทางการที่ต้อง map ให้ตรง
5. **ขนาด cohort ในเดโม่** (1 คณะโฟกัส vs หลายคณะ) — แนะนำหลายคณะเพื่อโชว์ analytics ข้ามคณะ

## 14. สคริปต์วันนำเสนอ (5 นาที)

1. **(30วิ) เปิดด้วยปัญหา:** "ตอนนี้กิจกรรมนิสิตกระจัดกระจาย ไม่มีภาพรวมว่าเด็กได้อะไรจริง"
2. **(1น) ฝั่งเจ้าหน้าที่:** สร้างกิจกรรมสด → ตั้งสไลเดอร์ 6 ด้าน → เห็นมินิเรดาร์ทรงคะแนน → ได้ QR
3. **(1น) โมเมนต์ wow:** ฉาย QR บนจอ → **ให้กรรมการสแกนด้วยมือถือจริง** → ชื่อกรรมการเด้งขึ้นจอสด ๆ
4. **(1.5น) พระเอก:** เปิด transcript ของนิสิตตัวอย่าง → เรดาร์วาดขึ้น → "คนนี้เด่นทักษะ+ความรู้ แต่ควรเสริมสุนทรียภาพ" → (optional) ฉายา
5. **(1น) ปิดด้วยคุณค่าเชิงระบบ:** dashboard กิจการนิสิต → "มหาลัยเห็นภาพรวมว่าคณะไหนจัดกิจกรรมด้านไหนน้อย ใช้วางแผนพัฒนานิสิตได้" → ปิดว่า "เฟสจริงต่อกับระบบทะเบียนกลางได้เลย ทีมเราเคยทำระบบมหาลัยมาก่อน"
